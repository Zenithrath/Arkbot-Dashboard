## CRITICAL: Actual Supabase Schema (from Phase 1)

The existing schema is designed for the WhatsApp bot, NOT a dashboard. Here's what actually exists:

### Existing Tables

**`documents`** — one row per chunk (NOT per file!)
| Column | Type | Notes |
|--------|------|-------|
| id | UUID | PK |
| content | TEXT | The chunk text content |
| metadata | JSONB | Contains file info (name, page, section, etc.) |
| embedding | vector | Gemini embedding |
| drive_file_id | TEXT | Google Drive file ID (links to drive_file_sync) |
| created_at | TIMESTAMPTZ | |

**`drive_file_sync`** — file-level sync tracking
| Column | Type | Notes |
|--------|------|-------|
| id | UUID | PK |
| drive_file_id | TEXT | Unique, links to documents.drive_file_id |
| file_name | TEXT | Original filename |
| last_modified_time | TIMESTAMPTZ | From Google Drive |
| last_synced_at | TIMESTAMPTZ | When last synced |
| status | TEXT | synced / pending / error |

**`chat_history`** — WhatsApp chat logs (not for dashboard)
**`whitelist_users`** — authorized phone numbers (not for dashboard)
**`processed_events`** — dedup tracking (not for dashboard)

### Key Insight
- **No activity_log table** — we need to create one OR derive activity from drive_file_sync status changes
- **No sync_status table** — sync info comes from drive_file_sync
- **documents = chunks** — each row IS a chunk. To get "document" info, we GROUP BY drive_file_id and read metadata JSONB
- **No chunk_count column** — we count rows per drive_file_id
- **No category/tags/folder columns** — these would live in metadata JSONB (if populated by n8n)
- **No document status column** — status is on drive_file_sync, not documents

### What the Dashboard Needs That Doesn't Exist
1. **`activity_log`** — must create this table for the Activity page
2. **`match_documents` RPC** — must create for Search Testing page
3. **Document metadata** (category, tags, folder) — either n8n populates `documents.metadata` with these, or we add columns. Plan assumes metadata JSONB has `{ name, category, tags, folder }` keys.

## Decision: Schema Extension

We need to create 1 Supabase table + 1 RPC function via SQL migration:

### New Table: `activity_log`
```sql
CREATE TABLE public.activity_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  type TEXT NOT NULL,           -- upload, delete, re_index, sync, failure, edit_metadata
  status TEXT NOT NULL,         -- success, error, pending
  description TEXT NOT NULL,
  drive_file_id TEXT,           -- links to drive_file_sync
  document_name TEXT,           -- denormalized for display
  "user" TEXT DEFAULT 'admin',
  metadata JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_activity_created_at ON activity_log(created_at DESC);
CREATE INDEX idx_activity_type ON activity_log(type);
```

### New RPC: `match_documents`
```sql
CREATE OR REPLACE FUNCTION match_documents(
  query_embedding vector,
  match_count INT DEFAULT 5,
  match_threshold FLOAT DEFAULT 0.5
)
RETURNS TABLE (
  chunk_id UUID,
  drive_file_id TEXT,
  document_name TEXT,
  chunk_content TEXT,
  similarity FLOAT
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    d.id AS chunk_id,
    d.drive_file_id,
    COALESCE(d.metadata->>'name', dfs.file_name, 'Unknown') AS document_name,
    d.content AS chunk_content,
    1 - (d.embedding <=> query_embedding) AS similarity
  FROM documents d
  LEFT JOIN drive_file_sync dfs ON dfs.drive_file_id = d.drive_file_id
  WHERE d.embedding IS NOT NULL
    AND 1 - (d.embedding <=> query_embedding) > match_threshold
  ORDER BY d.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

## Data Model Mapping (Dashboard ↔ Supabase)

| Dashboard Field | Source | How |
|----------------|--------|-----|
| Document Name | `drive_file_sync.file_name` | JOIN on drive_file_id |
| Category | `documents.metadata->>'category'` | JSONB field |
| Tags | `documents.metadata->>'tags'` | JSONB field (comma-separated or array) |
| Folder | `documents.metadata->>'folder'` | JSONB field |
| File Type | Derived from `file_name` extension | Parse `.pdf`, `.md`, etc. |
| Status | `drive_file_sync.status` | synced/pending/error |
| Chunk Count | COUNT of `documents` rows per drive_file_id | GROUP BY |
| Last Indexed | `drive_file_sync.last_synced_at` | Direct column |
| Last Modified | `drive_file_sync.last_modified_time` | Direct column |
| Google Drive ID | `drive_file_sync.drive_file_id` | Direct column |

### Document List Query Pattern
```sql
-- Get documents (aggregated from chunks)
SELECT
  dfs.id,
  dfs.drive_file_id,
  dfs.file_name,
  dfs.status,
  dfs.last_synced_at,
  dfs.last_modified_time,
  COUNT(d.id) as chunk_count,
  -- Extract metadata from first chunk
  (ARRAY_AGG(d.metadata))[1]->>'category' as category,
  (ARRAY_AGG(d.metadata))[1]->>'tags' as tags,
  (ARRAY_AGG(d.metadata))[1]->>'folder' as folder
FROM drive_file_sync dfs
LEFT JOIN documents d ON d.drive_file_id = dfs.drive_file_id
GROUP BY dfs.id, dfs.drive_file_id, dfs.file_name, dfs.status, dfs.last_synced_at, dfs.last_modified_time
ORDER BY dfs.last_synced_at DESC
```

## n8n Webhook Payloads

| Action | Payload |
|--------|---------|
| Upload | `{ action: "upload", file_name, category, tags, folder, file_base64, file_type, file_size }` |
| Re-index | `{ action: "re_index", drive_file_id }` |
| Delete | `{ action: "delete", drive_file_id, delete_from_drive: true }` |
| Sync | `{ action: "sync", force: false }` |

All return `{ success: boolean, message: string }`. Webhook URLs configurable in Settings (localStorage).
